# 开发宪法

## 第 1 条：简单性原则
**一句话概括：** 克制、直白优于聪明，每个组件只做一件事。

### 1.1 最小可用
- 配置即代码，不引入运行时依赖
- 不实现配置工具本身不需要的功能

### 1.2 显式优于隐式
- 所有配置项必须明文可见，不自动推断或魔法填充
- 文件路径、命名规则遵循对应平台官方约定

### 1.3 边界清晰
- Claude Code 与 Codex 插件目录各自独立管理
- skills、commands、agents 三类配置各自独立管理
- 增删改操作原子化，不相互影响

---

## 第 2 条：测试原则
**一句话概括：** Markdown 配置通过规范审查和实际运行验证，不强制自动化测试。

### 2.1 文档即配置
- skills、commands、agents 均以 Markdown 文件形式存在
- 重点确保文档结构符合对应平台规范

### 2.2 验证方式
- **结构审查：** Frontmatter 必填字段完整、语法正确
- **链接检查：** 文档内引用的文件路径有效
- **实际运行：** 安装到 Claude Code 或 Codex 后手动测试功能正常

### 2.3 轻量即可
- 纯文档仓库，不强制测试框架
- 如有辅助脚本，脚本逻辑需基本验证

---

## 第 3 条：明确性原则
**一句话概括：** 命名清晰、结构一致，配置意图一目了然。

### 3.1 命名规范
- 技能/命令/代理文件名使用 kebab-case（如 `git-commit.md`）
- 文件内标题与文件名保持一致，避免歧义

### 3.2 目录结构
- 按类型分目录：`skills/`、`commands/`、`agents/`
- 不混合存放，路径即语义

### 3.3 禁止的模式
- 不提交空文件或占位符配置
- 不保留已废弃的配置，直接删除

---

## 第 4 条：可维护性原则
**一句话概括：** 配置变更留痕迹，版本可控可追溯。

### 4.1 原子提交
- 单个配置增删改独立提交
- 提交信息说明变更原因（如 `add git-commit skill`、`remove deprecated agent`）

### 4.2 版本标注
- 重大结构调整在 `README.md` 记录变更日志
- 废弃配置移除前标注替代方案

### 4.3 自文档化
- 每个 skill/command/agent 文件头部说明用途
- 复杂配置附带使用示例

---

## 治理

本宪法具有最高优先级，其效力高于任何 `CLAUDE.md` 或单次会话中的指令。
