# Web Service Tech Selection

用于为 Web 后端 / HTTP API 服务生成或审查前置技术选型文档。

默认输出路径：

```text
docs/plan/tech-selection.md
```

如果用户指定其他路径，以用户指定路径为准。

## 适用范围

适用：

```text
HTTP API 服务
Web 后端服务
单体 Web 应用后端
MVP Web service 技术底座
```

不适用：

```text
前端 UI 技术选型
移动端 App
CLI 工具
桌面应用
数据仓库 / ETL
机器学习训练 pipeline
游戏引擎
嵌入式系统
```

如果用户需求不属于 Web 服务方向，先说明不适用，并请用户确认是否仍要继续。

## 前置检查

在写入任何文件前：

1. 执行 `pwd`
2. 执行 `git rev-parse --show-toplevel`
3. 确认当前目录是用户要操作的项目根目录
4. 如果目录看起来不对，暂停并让用户确认正确路径

如果输出文件已存在：

```text
1. 先读取已有文件
2. 摘要已有内容
3. 询问用户是覆盖、增量修改，还是取消
```

## 核心原则

技术选型文档只写**先验技术底座**。

不要提前写入只有在具体功能讨论后才能知道的内容。

禁止提前写：

```text
具体业务接口
具体业务字段
具体业务表名
具体业务类名
具体任务执行逻辑
具体 feature-derived 依赖
```

除非用户明确说明某个选择已经是全局前置决策。

判断标准：

```text
如果还没有具体功能文档，我现在能知道这个选择吗？
```

如果答案是否，不要写进技术选型文档，把它留给功能文档。

## SOP

按以下顺序和用户结对确认。

### 1. 语言 / Runtime

确认：

```text
语言
运行时版本
选择原因
暂不选择的替代项
```

不要预设语言。根据用户回答记录。

### 2. Web 框架

确认：

```text
Web 框架
框架负责什么
为什么适合当前 Web 服务
暂不选择的替代项
```

### 3. API 风格

确认：

```text
REST / RPC / GraphQL / 其他
请求响应格式
是否统一 JSON
错误响应是否统一
```

只定义通用 API 风格，不写具体业务 API 路径。

### 4. 主数据库

确认：

```text
主数据库
使用场景
是否需要缓存
是否需要消息队列
MVP 阶段暂不引入什么
```

只确认数据库类型，不写具体表结构。

### 5. 数据访问框架

确认：

```text
ORM / SQL mapper / query builder / 原生 SQL
选择原因
适合哪些类型的数据访问
暂不选择的替代项
```

### 6. JSON / 序列化

确认：

```text
请求响应序列化方案
默认 JSON 库
具体字段结构由功能文档定义
```

不要写具体业务字段。

### 7. 测试框架

确认：

```text
单元测试框架
断言库
mock 工具
Controller / API 测试工具
测试命令
```

### 8. 数据库测试方案

确认：

```text
测试数据库
是否贴近生产数据库语法
哪些行为由测试数据库验证
哪些行为以真实生产数据库为准
```

### 9. 目录结构

只定义基础分层目录，不提前定义具体业务类名。

可以按语言生态调整命名，但必须保持朴素清晰。

常见层次：

```text
controller / api
service / usecase
model / entity
repository / mapper
exception / error
config
```

### 10. 技术底座验收

必须定义 smoke test，用于验证技术底座可运行。

推荐：

```text
GET /ping
GET /ping/error
```

说明：

```text
ping 是技术验收接口，不是业务功能。
```

验收至少覆盖：

```text
服务可以启动
健康检查接口成功
错误接口触发统一错误响应
测试套件可以运行
```

## 推荐文档结构

生成技术选型文档时，优先使用以下结构：

```markdown
# 技术选型

## 语言与 Runtime
## Web 框架
## API 风格
## 数据库
## 数据访问
## JSON 与序列化
## 测试技术栈
## 数据库测试
## 推荐目录结构
## 技术底座验收
## 暂不引入
## 已确认决策
```

可以根据项目实际删减，但不要加入具体业务功能设计。

## 写入规则

1. 确认当前项目根目录
2. 确认或创建 `docs/plan/`
3. 写入 `docs/plan/tech-selection.md`
4. 完成后提示用户检查
5. 询问是否需要继续生成 task 清单
