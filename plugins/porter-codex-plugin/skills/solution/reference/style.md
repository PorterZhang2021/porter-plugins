# style — Solution 模板

## 类型专项分析必填字段

1. **目标** — 整理哪些风格问题，例如格式、命名、lint 规则。
2. **工具** — 使用哪些格式化工具、lint 工具或配置。
3. **影响范围** — 涉及哪些文件或目录。
4. **行为影响** — 是否改变业务行为；通常必须为"不改变"。
5. **验证方式** — 如何确认无报错且 diff 符合预期。

## 视觉模型

`style` 通常不强制 Mermaid。

- 如果 style 变更涉及格式化管线或 lint 流程，建议使用 `flowchart`。
- 单纯格式化或命名整理可以写明"无图，原因：..."。

## 验收标准写法

- 风格问题被整理。
- 业务行为不变。
- 工具执行无报错。
- diff 范围符合预期。

## 待确认建议

- 风格范围是否正确。
- 使用工具是否符合项目约定。
- 是否确认不改变业务行为。
- diff 范围是否需要限制。

## solution-task 提示

- 无业务逻辑，无需业务测试。
- 必须包含格式化、lint 或 diff 审查任务。
