# fix — Bug 修复规划

## 前置步骤：使用 bug-analyzer 分析问题

在生成 PLAN.md 前，如果用户提供了错误信息、堆栈跟踪或问题描述，**强烈建议使用 bug-analyzer agent 进行深度分析**：

```
使用 delegate_subtask 调用 bug-analyzer agent：
- 提供错误信息和堆栈跟踪
- 包含相关代码文件路径
- 描述期望行为 vs 实际行为
- Agent 将分析执行路径、根因和修复建议
```

如果没有足够的错误信息，询问用户是否需要先运行代码以重现问题。

## 生成 PLAN.md

生成 `plan/fix/<branch-name>/PLAN.md`，逐节与用户确认：

1. **Bug 描述** — 现象、触发条件、影响范围
   - 如果使用了 bug-analyzer，引用其分析结果

2. **复现路径** — 最小复现步骤
   - 具体的输入数据或操作序列

3. **根因分析** — 定位到具体代码位置和原因
   - 如果使用了 bug-analyzer，整合其根因分析
   - 标注具体的文件路径和行号

4. **修复方案** — 拟修改的文件和逻辑
   - 如果使用了 bug-analyzer，参考其修复建议
   - 说明为什么这个方案能解决问题

5. **回归测试** — 需要补充哪些测试用例防止复发
   - 至少包含一个能复现 Bug 的测试用例

## 收尾

提示运行 `/task` 生成修复任务清单（必须执行）

**重要**：第一个任务必须是"编写能复现 Bug 的失败测试"，遵循 TDD Red-Green-Refactor 流程。
